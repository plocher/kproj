# Phase 4 adversarial review
Scope: PRD/DESIGN/GLOSSARY/ADR review for kproj Phase 3 deliverables, with spot checks against the local SPCoast Jekyll site, local KiCad CLI, and jBOM API surface.
Counts: 6 BLOCKER, 10 MAJOR, 4 QUESTION, 2 MINOR.
## BLOCKER
### Direct iBOM script execution contradicts the locked jobset decision
Location: `docs/DESIGN.md:142`, `docs/DESIGN.md:155-161`, `docs/DESIGN.md:225-229`; plan `496d47fb-92d5-49d9-bbb0-5dea1bf0e99c` § Phase 1 Closeout / iBOM invocation.
Description: The locked plan says iBOM generation is `kicad-cli jobset run` against a kproj-generated `.kicad_jobset`; KiCad's job runner resolves the PCM-installed iBOM plugin, and kproj must not depend on the script path directly. DESIGN instead preflights `${KICAD9_3RD_PARTY}/.../generate_interactive_bom.py` and invokes `python <ibom-script> ...`. That is not an implementation detail; it changes the dependency boundary, test fixtures, failure mode, and Phase 5 issue shape.
Suggested resolution: Replace the direct Python invocation contract with a generated jobset contract: jobset file shape, `kicad-cli jobset run --file <jobset> --output <destination> <project.kicad_pro>`, expected output discovery, and failure diagnostics. Preflight should locate `kicad-cli` and optionally validate the jobset/plugin by running or dry-running KiCad's job runner, not by hard-coding the plugin script path.
### Site emission contract is aspirational, not consumable by the current Jekyll site
Location: `docs/DESIGN.md:354-380`; `docs/PRD.md:88-92`, `docs/PRD.md:198-204`; `/Users/jplocher/Dropbox/eagle/SPCoast.github.io/_layouts/eagle.html:58-90`, `/Users/jplocher/Dropbox/eagle/SPCoast.github.io/_layouts/eagle.html:205-226`, `/Users/jplocher/Dropbox/eagle/SPCoast.github.io/_versions/AD4DA-LightUI/2.2.md:1-39`.
Description: DESIGN emits `layout: kicad`, relative `image_path` / `artifacts[].path` values, and `audit` / `drc` / `erc` front-matter. The local site has no `_layouts/kicad.html`; existing versions use `layout: eagle` and absolute `/versions/...` asset paths. `eagle.html` renders image/artifact links literally and only renders version artifacts inside `{% if version.publish == true %}`. It also has no logic for the promised quality badge fields. A Phase 6 implementation following DESIGN will generate pages that do not render the promised assets or badges.
Suggested resolution: Either make the kproj contract match the current site (`layout: eagle`, absolute `/versions/<Project>/<R>/...` paths, required legacy fields such as `publish`, and a concrete status mapping), or add an explicit prerequisite site PR/issue that creates and validates `kicad.html`, updates `electronics.html`, and renders `audit`/`drc`/`erc`. Do not leave this as implicit Phase 7 site work.
### Asset generation escapes ChangeJournal rollback
Location: `docs/DESIGN.md:149-163`, `docs/DESIGN.md:314-320`; `docs/adr/0005-writetracker-transactional-site-writes.md:26-49`, `docs/adr/0005-writetracker-transactional-site-writes.md:75-86`.
Description: The pipeline has exporters write directly into `<site_repo>/versions/<P>/<R>/...` before `SitePublisher.publish()` starts, and `SitePublisher` says "assets are already in place." But ADR 0005 requires every created/modified site file to be tracked for rollback. As written, a `PcbExporter`, `SchematicExporter`, `IbomGenerator`, `FabPackager`, or `SourcePackager` failure can leave partial site files before the journal owns them, violating Story 9's clean rollback guarantee.
Suggested resolution: Start the ChangeJournal before any site-repo output is requested and pass it through all file-producing services, or stage every artifact outside the site repo and have `SitePublisher` journaled-move them into place. Specify how partially written subprocess outputs are registered and cleaned.
### Config layer leaks argparse outside `cli.py`
Location: `docs/DESIGN.md:81-90`, `docs/DESIGN.md:95-108`; `docs/adr/0006-library-shape-boundary-discipline.md:24-27`.
Description: ADR 0006 locks "argparse is confined to `src/kproj/cli.py`." DESIGN puts `load_config(cli_args: argparse.Namespace)` in `src/kproj/config.py`, forcing either an `argparse` import outside `cli.py` or a fragile unevaluated annotation. This breaks the library-shape boundary before implementation starts.
Suggested resolution: Have `cli.py` translate argparse output into a kproj-owned frozen dataclass, e.g. `ConfigOverrides(site_repo: Path | None, no_push: bool | None)`, and pass that into `load_config()`. Keep `argparse.Namespace` entirely inside `cli.py`.
### Project resolution contract misstates the jBOM API and returned artifact
Location: `docs/DESIGN.md:125-132`, `docs/DESIGN.md:197-202`; `/Users/jplocher/Dropbox/KiCad/jBOM/src/jbom/application/pcb_project_loader.py:66-113`.
Description: DESIGN specifies `KicadProjectReader.resolve(...) -> Path` returning the canonical `.kicad_pro` path, while the cited jBOM function returns a `ResolvedPcbProject` dataclass with `pcb_path`, `project_context`, and diagnostics. This is a concrete API mismatch, not a naming nit. A worker implementing the DESIGN signature will either discard diagnostics/context or treat a dataclass as a Path; downstream `read(project_path)`, `DesignAnalyzer`, and packagers are then ambiguous about whether they receive `.kicad_pro`, `.kicad_pcb`, or project directory.
Suggested resolution: Define a kproj-owned `ResolvedProject`/`ProjectPaths` dataclass that wraps jBOM's `ResolvedPcbProject` and explicitly carries `project_file`, `project_dir`, `pcb_file`, `root_schematic`, hierarchical schematics, and resolution diagnostics. Update all service signatures to accept that shape rather than a generic `Path`.
### FabPackager looks for the wrong production zip
Location: `docs/DESIGN.md:156`, `docs/DESIGN.md:237-242`, `docs/DESIGN.md:343-345`; `docs/adr/0003-jbom-separation-read-not-invoke.md:30-36`.
Description: ADR 0003 says jBOM's production gerber artifact is `<project_dir>/production/<title>_<rev>.zip`, and kproj repackages it inside `<P>-<R>.fab.zip` as `gerbers.zip`. DESIGN instead says FabPackager reads `production/{bom.csv, pos.csv, gerbers.zip}`. Unless jBOM changes its output name, kproj will falsely warn/omit fab artifacts for valid production directories.
Suggested resolution: Specify discovery of jBOM's gerber-pack zip by manifest if one exists, otherwise by a strict single-zip rule or known jBOM naming convention. The archive entry name in kproj's output can be normalized to `gerbers.zip`; the source file should not be required to have that name.
## MAJOR
### COMMENT9/status source side conflicts with the locked metadata precedence
Location: `docs/PRD.md:97-105`, `docs/DESIGN.md:197-202`; plan `496d47fb-92d5-49d9-bbb0-5dea1bf0e99c` § KiCad → Jekyll Front-Matter Contract.
Description: The locked plan says SCH is canonical for `comment2`, `comment3`, and `comment9`; PCB may routinely omit those fields. PRD Story 6 tells the user to edit `${COMMENT9}` in the PCB title-block, and DESIGN says `KicadProjectReader.read()` applies a broad "PCB-precedence rule for fields that differ between sides." That will miss or override the intended status source.
Suggested resolution: Update Story 6 to edit the SCH title block, or explicitly require both SCH and PCB edits. In DESIGN, replace the broad PCB-precedence wording with per-field precedence: PCB canonical/fallback only where locked, SCH canonical for status/tagline fields, and raw SCH/PCB values retained for audit findings.
### Private-skip semantics conflict with exit codes and preflight ordering
Location: `docs/PRD.md:109-117`, `docs/DESIGN.md:139-148`, `docs/DESIGN.md:392-397`.
Description: PRD says `status: private` runs audit + DRC/ERC, writes nothing, performs no git operations, and exits 0. DESIGN preflights site cleanliness and iBOM plugin availability before reading metadata/status, then maps `private-skip` with any findings to exit 1. That means a private project can fail because the site repo is dirty or iBOM is missing even though no site/iBOM writes should occur, and the promised exit code disagrees with the global findings-present contract.
Suggested resolution: Decide whether private skips are "successful with findings" (exit 1 per global contract) or a special exit-0 exception. Move private detection immediately after metadata read, before site cleanliness/iBOM/artifact preflights, unless the chosen semantics truly require those checks.
### Schematic SVG export treats an output directory as a file and omits root-sheet selection
Location: `docs/DESIGN.md:153`, `docs/DESIGN.md:214-222`; local `/Applications/KiCad/KiCad.app/Contents/MacOS/kicad-cli version` = `10.0.1`, `kicad-cli sch export svg --help` § Usage.
Description: DESIGN specifies `kicad-cli sch export svg --output <P>-<R>.sch.svg <root.kicad_sch>` and a service method returning exactly that file. The local KiCad CLI help says schematic SVG `--output` is `OUTPUT_DIR`, not `OUTPUT_FILE`, and supports `--pages PAGE_LIST`. Without an output-directory discovery/move step and explicit page selection, the implementation may create KiCad-named SVG(s) that do not match the front-matter asset reference or may export more than the root sheet.
Suggested resolution: Define the exact command as directory-based: create a temp output dir, pass `--pages <root-page>` if that is the correct KiCad selector, discover the generated SVG, then journal/move it to the site asset filename. Add a CLI-contract test around this behavior.
### New-release detection ignores body content, README content, asset presence, and asset freshness
Location: `docs/DESIGN.md:165-174`; `docs/PRD.md:48-57`, `docs/PRD.md:122-130`.
Description: DESIGN decides noop/refresh by comparing only parsed front-matter. PRD requires `pages/<Project>.md` to reflect README content and Story 13 promises no work only when already published with current state. A README change, changed audit table body, missing asset file, stale asset file, or changed source zip contents can all leave front-matter unchanged and still require publication work. The current detection would silently skip.
Suggested resolution: Define the comparison target as the complete rendered publication: front matter, body markdown, project page body, and expected asset manifest. At minimum, verify all referenced assets exist and use a deterministic content hash/mtime manifest for generated artifacts and source inputs before returning noop.
### Side-effect service result shapes drop diagnostics despite the Producer Pattern
Location: `docs/DESIGN.md:190-193`, `docs/DESIGN.md:206-246`, `docs/DESIGN.md:248-253`; `docs/GLOSSARY.md` § Producer Pattern.
Description: DESIGN says every service's primary method returns a typed result with `diagnostics: tuple[Finding, ...]`, but several side-effect services return only `Path`: `PcbExporter.export_render`, `PcbExporter.export_step`, `SchematicExporter.export_svg`, `SchematicExporter.export_pdf`, `IbomGenerator.generate`, `SourcePackager.package`, and `ZipArchiver.archive`. That erases warnings, command metadata, skipped/partial output states, and idempotence facts from the contract.
Suggested resolution: Either define result dataclasses for all producers, e.g. `ExportResult(path, diagnostics, command, duration, skipped=False)`, or explicitly document which services are intentionally exception-only and why they are exempt from the Producer Pattern.
### kicad-cli locator/version preflight is incomplete
Location: `docs/DESIGN.md:151-154`, `docs/DESIGN.md:522-523`; `/Users/jplocher/Dropbox/KiCad/jBOM/src/jbom/services/gerber_service.py:47-81`.
Description: The local shell has no `kicad-cli` on `PATH`, but the binary exists at `/Applications/KiCad/KiCad.app/Contents/MacOS/kicad-cli`. jBOM already has a private `_find_kicad_cli()` that checks this path, and the locked plan listed an upstream prerequisite to expose it. DESIGN examples invoke bare `kicad-cli` and do not specify the locator/version dependency, so Phase 6 can pass mocked tests and fail on the user's actual machine.
Suggested resolution: Make `KicadCliLocator` or imported `jbom.services.gerber_service.find_kicad_cli()` a concrete prerequisite and service dependency. Preflight should report the discovered executable and version, enforce the supported major version, and all subprocess services should use the resolved executable path.
### Source archive cannot promise self-contained KiCad opening while excluding external libraries
Location: `docs/PRD.md:220-226`, `docs/DESIGN.md:252-254`; plan `496d47fb-92d5-49d9-bbb0-5dea1bf0e99c` § Source snapshot contents.
Description: PRD requires `source.zip` contents to open in KiCad without modification. The locked source snapshot explicitly excludes the shared `~/Dropbox/KiCad/SPCoast_KiCad_Library`, and DESIGN only includes project-local KiCad files plus library tables. A consumer without the user's global/shared library setup will likely open the project with missing symbols/footprints/3D models.
Suggested resolution: Either weaken the acceptance criterion to "contains the non-derived project sources plus documented external library requirements," or add a dependency manifest/site artifact that tells consumers how to install the SPCoast KiCad library. Do not promise a self-contained open unless kproj actually vendors or packages the required libraries.
### Subprocess timeout, hang, and signal behavior is unspecified
Location: `docs/DESIGN.md:526-535`, `docs/adr/0005-writetracker-transactional-site-writes.md:75-86`.
Description: DESIGN handles non-zero subprocess returns and filesystem exceptions, but not `TimeoutExpired`, hung KiCad/iBOM processes, `KeyboardInterrupt`, `SIGTERM`, or partial output after termination. Batch safety depends on these paths: a hung subprocess prevents the batch from progressing, and a signal after site writes may bypass rollback unless the journal scope and top-level handler are precise.
Suggested resolution: Specify a shared subprocess runner with per-step timeouts, captured stdout/stderr, cancellation behavior, and rollback on `BaseException` types that should be caught. Document which signals can be rolled back and which require manual recovery.
### Test strategy can validate mocks instead of external contracts
Location: `docs/DESIGN.md:452-507`.
Description: The testing plan says subprocess-heavy services are mocked in unit tests and real `kicad-cli` integration is validated only in Behave scenarios against fixture projects. The fixture list does not state that Behave actually runs local KiCad, validates KiCad's generated filenames/JSON schema, or validates the iBOM jobset contract. This leaves the exact class of Phase 6 risks in DESIGN (wrong CLI output shape, wrong script/jobset contract, wrong JSON parser) able to pass tests.
Suggested resolution: Add contract tests that run the discovered local `kicad-cli` on tiny fixtures for DRC/ERC JSON, schematic SVG/PDF export, PCB render/STEP export, and jobset/iBOM behavior where available. Unit mocks should assert command construction; integration tests should assert real output discovery and parser shape.
### Audit exception rules are not implementable enough to avoid false positives and false negatives
Location: `docs/DESIGN.md:334-345`; `docs/GLOSSARY.md` § design_rev / board_rev.
Description: The audit list says `sch_pcb_disagree` excludes legitimate rev/date patterns "documented in CONTEXT history" and `rev_relation` only checks whether `pcb_rev` starts with `sch_rev`. That is under-specified for implementation and too permissive: `3.0-beta`, `3.0.1`, or `3.0B2` all start with `3.0` but do not match the locked "design revision plus letter suffix" domain language. The date exception also lacks a concrete "small interval" threshold and direction rule.
Suggested resolution: Put the executable rules in DESIGN: e.g. `board_rev` must match `^<escaped design_rev>[A-Z]*$` after normalization, and `pcb_date` may follow `sch_date` by at most N months/days. Add examples for accepted and rejected cases.
## QUESTION
### Is `--site-repo` accepted or rejected?
Location: `docs/adr/0007-local-cli-v1-ci-deferred.md:20`, `docs/DESIGN.md:84-90`, `docs/PRD.md:253`.
Description: ADR 0007 says the site repo path can come from a `--site-repo` flag, but PRD/DESIGN and the locked CLI surface list only the positional, `--dry-run`, `--no-push`, `-v`, and `-d`. Phase 5 needs to know whether to cut CLI/config work for `--site-repo` or treat ADR 0007's mention as stale.
Suggested resolution: Architect should choose one. If the locked minimal CLI wins, remove `--site-repo` from ADR 0007 or supersede that sentence; if the ADR wins, update PRD/DESIGN and the locked CLI list.
### What should `private` do to an already published version?
Location: `docs/PRD.md:109-117`, `docs/DESIGN.md:144-148`, `docs/DESIGN.md:170-174`.
Description: The docs say private skips site writes. If a version was previously public and `${COMMENT9}` changes to `private`, skipping all writes leaves the old version visible on the site. If privacy means "do not publish future snapshots only," that is fine; if it means "make this release not public," the current design fails.
Suggested resolution: Clarify whether private is prospective-only or should hide/remove/update existing site entries. If it hides, define whether kproj deletes files, flips `publish`, changes `iskicad`, or leaves a tombstone.
### What is the "standard asset set" when `production/` is missing?
Location: `docs/PRD.md:48-57`, `docs/PRD.md:214-218`; `docs/adr/0003-jbom-separation-read-not-invoke.md:58-62`.
Description: Story 1 requires the standard asset set, and Story 16 assumes `fab.zip` exists and contains exactly three files. ADR 0003 says missing/empty `production/` is only a warning and `fab.zip` is omitted. Those can coexist only if the story GIVEN clauses include fresh production outputs or if "standard asset set" is explicitly conditional.
Suggested resolution: Add story preconditions for fresh jBOM production outputs where fab artifacts are required, and define the artifact list when `production/` is absent/stale.
### When does `comment9_missing` become an error, and does empty still default to `active`?
Location: `docs/DESIGN.md:334-336`; plan `496d47fb-92d5-49d9-bbb0-5dea1bf0e99c` § KiCad → Jekyll Front-Matter Contract.
Description: DESIGN labels missing `${COMMENT9}` an error "after corpus bulk-populate" but says to treat it as warning during transition. The locked plan says absent/empty `${COMMENT9}` defaults to `active`. Phase 5 cannot write stable tests without knowing the date/condition for the severity switch and whether the default is still emitted even when a finding exists.
Suggested resolution: Make the v1 rule explicit: likely "empty status emits `active` and creates a warning" unless/until a future ADR changes policy.
## MINOR
### `--dry-run` is not milliseconds if it runs DRC/ERC
Location: `docs/DESIGN.md:403-412`; `docs/PRD.md:62-66`.
Description: DESIGN says dry-run runs `DesignAnalyzer` and logs DRC/ERC findings, then says dry-run is "fast (milliseconds)." Real KiCad DRC/ERC can take seconds or longer and writes report files to temp paths. This is not fatal, but the UX promise is wrong.
Suggested resolution: Remove the milliseconds claim, or split dry-run into "metadata-only preview" versus "full analysis preview" if speed is a requirement.
