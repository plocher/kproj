## BLOCKER
### Production artifact paths use the project name as the board revision
Location: `src/kproj/application/publish_workflow.py:519`, `src/kproj/application/publish_workflow.py:524-590`
Description: `_default_artifact_generator()` does not receive `ProjectInfo` or `board_rev`, so it sets `board_rev_approx = resolved.project_file.stem`, which is the project basename, not the PCB revision. A real project `MyProject` with board revision `1.0B` will generate assets under `versions/MyProject/MyProject/` named `MyProject-MyProject.*`; the version file is still `_versions/MyProject/1.0B.md`. This breaks the locked `<site_repo>/versions/<Project>/<board_rev>/<P>-<R>.*` asset contract and also passes the wrong `rev` into `FabPackager`, so canonical jBOM gerber discovery can miss valid `<title>_<board_rev>.zip` files. Unit and Behave tests do not catch this because they inject an artifact-generator stub that hard-codes `R = "1.0"`.
Suggested resolution: Pass `ProjectInfo` or the canonical `board_rev` into the artifact-generator interface and remove the heuristic entirely. Add a test where `project_file.stem != board_rev` and assert both on-disk asset paths and emitted `AssetRef.path` values use the board revision.
### Generated assets are never staged or committed
Location: `src/kproj/services/site_publisher.py:190-196`, `src/kproj/services/change_journal.py:137-144`
Description: Producer services register artifact outputs with `ChangeJournal`, but `SitePublisher.publish()` stages only the version markdown and project page: `git add _versions/<P>/<R>.md pages/<P>.md`. The generated `versions/<P>/<R>/*` files are left untracked and unpushed while the committed markdown links to them. This violates PRD Story 1's "standard asset set" commit/push expectation and undercuts ADR 0005's design that `journal.all_paths()` is the tracked publish set. It also poisons batch runs: after a "successful" publish the site repo still has untracked asset files.
Suggested resolution: Stage every path registered in the journal, not just markdown files. Use `journal.all_paths()` for `git add`, and add an end-to-end-ish unit test with a real temporary git repo asserting `git status --porcelain` is clean and all asset files are present in `HEAD`.
### Rollback deletes previously published assets because producers always journal outputs as creations
Location: `src/kproj/services/pcb_exporter.py:174`, `src/kproj/services/schematic_exporter.py:108`, `src/kproj/services/schematic_exporter.py:169`, `src/kproj/services/ibom_generator.py:111`, `src/kproj/services/fab_packager.py:173`, `src/kproj/services/source_packager.py:142`
Description: Every artifact producer calls `journal.will_create(output)` without checking whether the output already exists. On a re-publish of a stale existing version, the first producer can overwrite an already committed asset and a later producer can fail. Rollback then unlinks the "created" path instead of restoring the prior tracked version, leaving the site repo missing previously published assets and violating ADR 0005's guarantee that modified files are restored to the pre-kproj state.
Suggested resolution: Producers must use `will_modify(output)` when `output.exists()` before writing and `will_create(output)` only for genuinely new outputs. Add rollback tests that start from a committed existing asset, force a failure after one overwrite, and assert the original bytes and clean git status are restored.
### Findings are not printed to CLI stderr
Location: `src/kproj/cli.py:195-200`, `src/kproj/formatters/stderr_formatter.py:43-65`, `src/kproj/application/publish_workflow.py:220`
Description: ADR 0004 locks that every audit/DRC/ERC finding is surfaced on stderr. PRD Story 2 also requires dry-run stderr to list findings and would-be writes. The CLI only prints `result.message`; it never instantiates or calls `StderrFormatter.format_findings(result.findings)`. Findings can set exit code 1 and render into the version file, but they are absent from the user's terminal output. The workflow independently prints the kicad-cli version line directly to stderr, so the only always-visible default stderr output is not the finding stream required by the ADR.
Suggested resolution: Move stderr rendering responsibility into `cli.py`: print formatted findings at default verbosity, then the result message if useful. Thread `produced_paths` / would-be paths into `PublishResult` for dry-run, and add CLI-level tests that capture stderr for audit warnings, DRC/ERC findings, dry-run paths, and no-finding/no-op silence.
### Schematic export shape errors escape the workflow as uncaught crashes
Location: `src/kproj/services/schematic_exporter.py:42-48`, `src/kproj/application/publish_workflow.py:329`
Description: `SchematicExporter` raises `SchematicExportError` when kicad-cli produces zero SVGs or multiple root-only SVGs. The workflow catch block only handles `SubprocessFailedError`, `SubprocessTimeoutError`, and `OSError`. A real output-shape mismatch therefore rolls back via `ChangeJournal.__exit__`, then escapes `PublishWorkflow.run()` and the CLI as a Python traceback instead of `PublishResult(outcome="failed", exit_code=2)` with a clear stderr message. This contradicts DESIGN's error handling contract for schematic export failures.
Suggested resolution: Catch `SchematicExportError` (or a common kproj pipeline exception base) in `PublishWorkflow.run()` and convert it to a mechanical failure result. Add a test where `SchematicExporter.export_svg()` raises `SchematicExportError` after an earlier artifact write and assert rollback plus exit code 2.
## MAJOR
### New-release detection ignores asset freshness
Location: `src/kproj/services/site_publisher.py:236-269`
Description: DESIGN § New-release detection requires comparing each asset's mtime against its corresponding source: PCB for renders/STEP/iBOM, schematic for SVG/PDF, project source set for source.zip, and `production/` for fab.zip. `detect_outcome()` only checks that referenced assets exist, then compares version markdown and project page content. If the PCB changes but front matter is stable, kproj can return `noop` or `refresh` and leave stale renders, iBOM, STEP, fab, and source archives on the site.
Suggested resolution: Model source freshness explicitly, either by adding source paths/mtimes to `Publication` or by moving detection into `PublishWorkflow` where `ResolvedProject` is available. Add tests where an existing asset is older than its source and assert `outcome == "publish"`.
### Front-matter audit/drc/erc counts are not source-specific
Location: `src/kproj/formatters/front_matter_summary_formatter.py:128-151`, `src/kproj/model/analysis_info.py:17-34`
Description: `AnalysisInfo` is a single merged findings tuple with no source dimension. `render_audit()` counts all `Severity.ERROR` and `Severity.WARNING` findings, including DRC/ERC. `_count_design_findings()` ignores `kind` and emits the same merged counts for both `drc:` and `erc:`. A single DRC error will appear as `audit.errors: 1`, `drc.errors: 1`, and `erc.errors: 1`, contradicting PRD Story 5 and the front-matter contract.
Suggested resolution: Preserve finding origin as data, not formatter inference. Options: split `AnalysisInfo` into audit/drc/erc collections, or add a `source`/`category` field to `Finding`. Count each front-matter block from its own source.
### DRC/ERC Markdown tables drop the actual KiCad location
Location: `src/kproj/services/design_analyzer.py:182-204`, `src/kproj/formatters/markdown_table_formatter.py:111-118`
Description: `DesignAnalyzer` stores the KiCad location (`pos`, `uuid`, `description`, or `sheet`) in `Finding.value` and stores the source label (`"drc"` / `"erc"`) in `Finding.location_hint`. `MarkdownTableFormatter` renders the DRC/ERC "Location" column from `location_hint`, so published findings will show `drc` or `erc` in every location cell instead of the coordinate or item identifier. The unit test masks this by constructing fake DRC findings with `location_hint="(50, 75)"`, which is not what `DesignAnalyzer` actually emits.
Suggested resolution: Render the Location column from `Finding.value` and add a separate Source column if `drc`/`erc` needs to be visible. Add a formatter test using findings produced by `DesignAnalyzer` rather than a hand-shaped fake.
### DesignAnalyzer treats missing DRC/ERC JSON as "no findings"
Location: `src/kproj/services/design_analyzer.py:144-151`, `tests/unit/services/test_design_analyzer.py:131`
Description: The analyzer calls kicad-cli with `check=False` because KiCad may return nonzero when violations exist. It then ignores the returned `returncode`. If kicad-cli fails mechanically and writes no JSON, `_run_kicad_subcommand()` returns `()` and the publish continues as if DRC/ERC were clean. The unit test explicitly locks this behavior with "When kicad-cli writes nothing, the analyzer returns no findings", which is the opposite of the mechanical-failure contract.
Suggested resolution: Capture the `SubprocessResult`; if no JSON is written, treat any nonzero return or stderr-bearing failure as a mechanical failure. Only convert nonzero violation exits into findings when parseable JSON exists. Add contract tests for real kicad-cli DRC/ERC return codes.
